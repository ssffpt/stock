## 详细审查结果

### 1. 后端审查

#### 1.1 models.py - ClearedPosition 模型

**问题 1：字段类型与 PRD 不一致**

| 字段 | PRD/Schema 定义 | 模型实际定义 | 问题 |
|------|----------------|--------------|------|
| stock_name | VARCHAR(50) | VARCHAR(100) | 长度不一致 |
| total_buy_qty | INT | DECIMAL(15,2) | 类型不一致 |
| profit_rate | DECIMAL(10,2) | DECIMAL(10,4) | 精度不一致 |

**问题 2：缺少 avg_sell_price 计算**

`calculate_cycle_stats()` 函数返回的字典中没有 `avg_sell_price` 字段（176-253行），但 `ClearedPosition` 模型要求此字段。upsert 时使用默认值 `Decimal("0")`，这会导致卖出均价为0的异常数据。

#### 1.2 crud.py - 清洗逻辑

**严重问题 1：cycle_index 递增逻辑错误（707行）**

```python
existing_count = db.query(ClearedPosition).filter(
    ClearedPosition.stock_code == stock_code
).count()
cycles = _get_stock_cycles_from_deliveries(db, stock_code, existing_count + 1)
```

**问题分析**：
- 第一次清洗股票 A：existing_count=0，生成 cycle_index 1,2,3
- 第二次清洗股票 A（数据无变化）：existing_count=3，生成 cycle_index 4,5,6
- 结果：预计算表中有6条记录，但实际上只有3个周期，数据重复

**预期行为**：根据 PRD "覆盖策略说明"，清洗应该覆盖/更新已有周期，而不是新增。

**严重问题 2：upsert 未正确实现覆盖**

`_upsert_cleared_positions()` 函数（585-646行）查询现有记录后逐个更新/插入，但：
- 没有先删除该股票所有旧记录
- cycle_index 基于 existing_count 递增，会导致重复周期

**问题 3：calculate_cycle_stats 缺少 avg_sell_price**

该函数没有计算 `avg_sell_price`，导致 upsert 时此字段为 0。

#### 1.3 schemas.py

无重大问题，Schema 定义与 API 设计基本一致。

#### 1.4 routers/cleared_position.py

**问题 1：CleanResponse 返回格式不一致**

API 设计（API设计.md）返回：
```json
{
  "code": 200,
  "data": {
    "success": true,
    "cleaned_count": 120,
    "cycle_count": 365,
    "message": "..."
  }
}
```

实际返回（94-110行）：
```json
{
  "success": true,
  "cleaned_count": 120,
  "message": "..."
}
```

缺少 `cycle_count` 字段，且外层缺少 `code` 包装。

**问题 2：CleanResponse 的 success 判断逻辑不够严谨（107行）**

```python
success=cleaned_count > 0 or "成功" in message,
```

这种判断可能在误判情况（如数据库错误返回包含"成功"的消息）。

## 修复建议

### 1. 修复 cycle_index 逻辑（优先级：高）

```python
def clean_cleared_positions(
    db: Session,
    start_date: Optional[date] = None,
    end_date: Optional[date] = None,
    stock_codes: Optional[List[str]] = None,
) -> Tuple[int, str]:
    # ... 确定 target_stocks ...

    # 修复：先删除这些股票的旧记录，再重新计算插入
    for stock_code in target_stocks:
        # 删除该股票的旧记录
        db.query(ClearedPosition).filter(
            ClearedPosition.stock_code == stock_code
        ).delete()

    # 重新计算所有股票的周期（从1开始）
    for stock_code in target_stocks:
        cycles = _get_stock_cycles_from_deliveries(db, stock_code, 1)
        if cycles:
            _upsert_cleared_positions(db, cycles)
            total_cleaned += len(cycles)

    db.commit()
    return total_cleaned, f"成功清洗 {total_cleaned} 个周期"
```

### 2. 修复 avg_sell_price 计算（优先级：高）

在 `calculate_cycle_stats()` 中添加：

```python
# 卖出均价
total_sell_qty = sum(r['quantity'] for r in sell_records)
if total_sell_qty > 0:
    avg_sell_price = (total_sell_amount / total_sell_qty).quantize(Decimal("0.001"))
else:
    avg_sell_price = Decimal("0")
```

### 3. 统一字段定义（优先级：中）

修正 models.py 以匹配 PRD：
- `stock_name`: VARCHAR(50)
- `total_buy_qty`: Integer（或保持 DECIMAL 但与 schema 一致）
- `profit_rate`: DECIMAL(10,2)

### 4. 修复前端默认日期（优先级：低）

在 CleanControl.tsx 中实现"本月"默认日期。

## 总结

本次代码审查发现了**2个严重问题**和**多个中等问题**：

1. **cycle_index 递增逻辑错误**会导致预计算表数据膨胀，每次清洗都会新增周期而不是覆盖
2. **avg_sell_price 缺失**会导致卖出均价数据始终为0

建议修复上述严重问题后再进行集成测试。
