# 清仓数据预计算功能 - 产品需求文档

## 1. 功能概述

### 问题背景
现有系统每次查询清仓数据时，都需要扫描全部交割记录并重新计算。大数据量时（如数万条交割单），查询响应慢，影响用户体验。

### 解决方案
新建 `cleared_position` 表存储预计算数据，采用增量更新策略。查询时直接读取预计算结果，避免重复计算。

---

## 2. 用户故事

### 场景一：日常查询
用户打开清仓数据页面，系统秒级展示历史清仓记录，无需等待计算。

### 场景二：数据修正后刷新
用户在导入新的交割单数据后，需要手动触发清洗，系统重新计算相关股票的清仓数据并更新预计算表。

### 场景三：历史数据回溯
用户需要重新计算某一时间段内的清仓数据，可选择日期范围进行增量清洗。

---

## 3. 功能详情

### 3.1 手动清洗按钮
- 位于页面顶部操作区
- 点击后触发增量计算，将符合条件的的数据写入 cleared_position 表
- 清洗过程中按钮显示 loading 状态，禁止重复点击
- 清洗完成后显示成功提示

### 3.2 时间段选择器
- 支持选择增量清洗的时间范围
- 组件：两个日期输入框（开始日期、结束日期）+ 快捷选项（本月、近三月、自定义）
- 默认选中：本月
- 时间范围用于筛选需要重新计算的交割单

### 3.3 股票选择器
- 支持按股票代码筛选需要清洗的股票
- 支持多选（逗号分隔或下拉多选）
- 为空时表示清洗所有有变化的股票

### 3.4 清洗进度提示
- 实时显示清洗进度：「正在清洗... 50/120 股票」
- 完成后显示统计：「清洗完成，共处理 120 支股票，360 个周期」

### 3.5 清洗历史/日志
- 记录每次清洗的时间、范围、结果
- 显示最近 10 条清洗记录
- 日志字段：清洗时间、日期范围、清洗股票数、周期数、操作状态

### 3.6 周期笔记功能
- 支持为每个清仓周期添加Markdown格式的笔记
- 笔记内容存储在cleared_position表的notes字段
- 支持编辑和保存操作
- 笔记内容可包含：
  - 交易复盘总结
  - 盈亏原因分析
  - 操作反思记录
  - 其他需要备注的信息

## 4. 新表结构 cleared_position

### 表名
`cleared_position`

### 字段设计

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | BIGINT | 主键，自增 |
| stock_code | VARCHAR(20) | 股票代码 |
| stock_name | VARCHAR(50) | 股票名称 |
| cycle_index | INT | 周期序号（同一股票的第几个完整买卖周期） |
| open_date | DATE | 建仓日期 |
| close_date | DATE | 清仓日期 |
| hold_days | INT | 持仓天数 |
| total_buy_amount | DECIMAL(15,2) | 买入总金额 |
| total_sell_amount | DECIMAL(15,2) | 卖出总金额 |
| total_buy_qty | INT | 买入总数量 |
| profit_loss | DECIMAL(15,2) | 盈亏金额 |
| profit_rate | DECIMAL(10,2) | 收益率(%) |
| avg_buy_price | DECIMAL(10,3) | 买入均价 |
| avg_sell_price | DECIMAL(10,3) | 卖出均价 |
| record_ids | TEXT | 对应原始交割单ID列表，JSON格式 |
| notes | TEXT | Markdown格式的周期笔记 |
| updated_at | DATETIME | 最后更新时间 |

### 索引设计
- 主键索引：id
- 联合唯一索引：uk_stock_cycle (stock_code, cycle_index)
- 普通索引：idx_close_date (close_date)

## 5. 增量清洗逻辑说明

### 5.1 如何判断哪些股票需要重新计算

1. **有新的交割单记录**：扫描 original_delivery 表，比较每条记录的 created_at 与 cleared_position 表中对应股票的最新 updated_at
2. **指定股票代码**：用户明确指定股票代码时，只计算这些股票
3. **指定时间范围**：用户选择日期范围时，只计算该范围内有变动的股票

### 5.2 时间段选择如何影响清洗范围

| 场景 | 清洗范围 |
|------|----------|
| 不指定时间范围 | 所有有变化的股票（全局增量） |
| 指定时间范围 | 该时间范围内有变化的股票 |
| 指定股票 + 指定时间 | 只清洗这些股票在该时间范围内的数据 |

### 5.3 覆盖操作说明

- **覆盖策略**：使用 `REPLACE INTO` 或 `ON DUPLICATE KEY UPDATE`
- **唯一键**：(stock_code, cycle_index)
- **行为**：如果某股票某周期已存在预计算数据，则覆盖更新；不存在则插入
- **原子性**：整个清洗过程在事务中执行，保证数据一致性

### 5.4 清洗流程

```
1. 解析清洗参数（时间范围、股票列表）
2. 查询原始交割单记录（按参数筛选）
3. 按股票代码分组
4. 对每只股票执行周期切割（复用 split_into_cycles）
5. 对每个完整周期计算统计（复用 calculate_cycle_stats）
6. 关联行情数据计算成本差
7. 写入/更新 cleared_position 表
8. 记录清洗日志
```

---

## 6. 收益与约束

### 收益
- 查询响应时间从数秒降至毫秒级
- 支持更大的数据量和更多并发查询
- 清洗操作可异步执行，不阻塞前端

### 约束
- 清洗操作需要一定时间（大数据量时可达分钟级）
- 预计算表与原始数据需要保持一致，清洗后需验证
- 历史数据修正后必须手动触发清洗
